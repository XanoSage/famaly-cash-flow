# XLSX Parser Implementation

Документ описывает первый технический слой импорта банковской выписки.

## Что уже делает парсер

Модуль: `backend/app/importers/bank_xlsx.py`.

Парсер читает XLSX-выписку банка и возвращает нормализованный preview-результат:

- имя исходного файла;
- название листа;
- период выписки;
- список строк для будущей таблицы `import_preview_rows`;
- summary по строкам, ошибкам, накоплениям, FOP-кандидатам и версиям правил.

## Почему парсер ищет колонки по названиям

Выписка сейчас имеет лист `Виписки`, период в первой строке, заголовки во второй строке и данные с третьей строки. Но в коде мы не привязываемся жестко к буквам колонок `A`, `B`, `C`.

Причина: если банк поменяет порядок колонок или добавит пустую строку сверху, поиск по названиям заголовков переживет это изменение лучше.

## Какие поля нормализуются

Для каждой операции парсер возвращает:

- дату операции;
- сумму в валюте карты;
- валюту карты;
- сумму и валюту оригинальной транзакции;
- баланс после операции;
- карту или payment instrument;
- банковскую категорию;
- описание операции;
- первичный merchant name;
- suggested flow type;
- suggested scope;
- статус preview;
- reason codes;
- normalized payload.

## Первичные правила

Сейчас правила простые и специально прозрачные:

- `Скарбничка` и `Заощадження` -> `transfer_to_savings`;
- `Зняття готівки` -> `cash_withdrawal`;
- `Переказ на свою картку` и `Зарахування зі своєї картки` -> `transfer_to_own_account`;
- `Перекази` -> `person_transfer` и `needs_review`;
- `Платежі за реквізитами` -> `requisites_payment` и `needs_review`;
- крупные операции от `5000 UAH` -> `large_amount`;
- крупный супермаркет от `2500 UAH` -> `large_supermarket`;
- признаки `fop`, `фоп`, `github`, `openai`, `upwork` -> `work_fop`.

Эти правила не финальные. Они нужны, чтобы первый preview уже отделял очевидные случаи от строк, где лучше спросить пользователя.
