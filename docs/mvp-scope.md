# MVP Scope

## Входит в MVP

### Пользователи и доступ

- Два отдельных пользователя в одной семье.
- Email/password авторизация.
- Access token + refresh token.
- Оба пользователя видят общий семейный бюджет.

### Источники данных

- Импорт XLSX-выписки одного банка.
- Один основной банковский источник.
- Основной счет в UAH.
- Физическая и виртуальная карта одного счета объединяются в один денежный поток.
- Маска карты доступна как фильтр в списке операций, но не выделяется в отдельную аналитику дашборда.
- Импорт сохраняет валюту оригинальной транзакции, если банк ее отдает.
- Ручное добавление наличных операций.

### Импорт

- Загрузка XLSX.
- Preview перед импортом.
- Предложение категорий и продавцов.
- Возможность исправить категорию, подкатегорию и продавца.
- Проверка дублей.
- Подтверждение импорта.
- Исходный XLSX после обработки не хранится.
- Метаданные импорта сохраняются.

### Операции

- Просмотр списка операций.
- Фильтры по периоду, категории, продавцу, типу операции.
- Редактирование операции.
- Scope операции: семейная, личная основного пользователя, рабочая/FOP.
- Для входящих операций: доход, возврат, перевод между своими счетами, долг, другое.
- Рабочие/FOP операции скрываются из семейной аналитики по умолчанию, но доступны отдельным блоком и фильтром.

### Категории

- Двухуровневые категории: категория + подкатегория.
- Стартовый набор категорий.
- Возможность добавлять свои категории.
- Правила автокатегоризации.
- Банковские категории используются как подсказки и маппятся на собственную структуру категорий.
- Уверенные маппинги применяются автоматически, сомнительные операции идут на проверку.
- Полная стартовая структура категорий описана в [Category Taxonomy](category-taxonomy.md).

### Продавцы

- Извлечение продавца или места покупки из выписки.
- Статистика по продавцам.
- Правило продавец -> категория.

### Наличные

- Снятие наличных с карты считается переводом карта -> наличный кошелек.
- Наличные расходы добавляются вручную.
- Баланс наличных приблизительный, потому что часть расходов может быть забыта.

### Накопления

- `Скарбничка`, округления и 1% от доходов считаются накоплениями.
- Накопления не смешиваются с обычными расходами.
- На дашборде нужен отдельный блок накоплений: сумма за период, динамика, сравнение с расходами, прогноз за год.

### Переводы

- Переводы физлицам отправляются на проверку.
- Получатель сохраняется как contact/merchant.
- Можно создать правило получатель -> категория/scope.
- Переводы жене учитываются, потому что это деньги из семейного потока.
- По умолчанию перевод жене можно показывать как "Перевод жене", но пользователь может уточнить: компенсация семейной траты или личные расходы жены.

### Платежи по реквизитам

- Первый платеж по реквизитам отправляется на проверку.
- Получатель сохраняется как merchant/contact.
- После ручной классификации создается правило получатель -> категория.
- Примеры: школа сына -> Дети / Образование; футбол сына -> Дети / Спорт.

### Аналитика

- Сводка за период.
- Расходы по категориям и подкатегориям.
- Динамика расходов по дням, неделям, месяцам.
- Топ продавцов и мест покупок.
- Накопления за период и динамика накоплений.
- Общий месячный бюджет и лимиты по важным категориям.
- Внутренние веб-уведомления о лимитах и операциях, требующих проверки.
- Простые фактические инсайты.

### Import Review

Операции отправляются на проверку, если:

- это перевод физлицу или "на картку";
- это платеж по реквизитам без правила получателя;
- операция больше лимита крупной операции;
- продавец новый и нет правила;
- уверенность категоризации низкая;
- операция потенциально рабочая/FOP.

Лимит крупной операции настраиваемый, значение по умолчанию: 5000 UAH.
Лимит крупного супермаркетного чека настраиваемый, значение по умолчанию: 2500 UAH.

Preview импорта должен поддерживать:

- таблицу всех операций;
- фильтр "требуют проверки";
- фильтры по дублям, переводам, крупным операциям, новым продавцам;
- фильтр по продавцу и банковской категории;
- массовое назначение категории одинаковому продавцу;
- предложение применить исправление к похожим операциям текущего импорта;
- предложение сохранить правило для будущих импортов.

### Подписки

- Подписки и цифровые сервисы учитываются отдельной категорией уже в MVP.
- Повторяющиеся платежи по продавцу и похожей сумме используются для будущих инсайтов.
- Отдельный экран "Подписки" переносится на будущую версию.

### Доходы

- Переводы между своими картами считаются внутренними переводами.
- Возвраты отделяются от доходов.
- Известные регулярные источники дохода могут классифицироваться правилом.
- Неизвестные входящие операции идут на проверку.

### Бюджеты

- В MVP нужен общий месячный семейный бюджет.
- Также нужны лимиты по важным категориям.
- Первый черновик бюджета предлагается на основе первой загруженной выписки и помечается как предварительный.

### Telegram Later

Будущая версия Telegram-бота:

- быстрый ввод наличных расходов;
- напоминания после снятия наличных;
- короткая сводка за день/неделю/месяц;
- уведомления о лимитах.

### Локализация

- Интерфейс на русском и украинском.
- Отчеты на русском и украинском.
- Язык выбирается на уровне пользователя.

## Не входит в первый MVP

- Подключение к банковскому API.
- Импорт PDF и скриншотов.
- Полный учет карт жены.
- Полноценная мультивалютность счетов и бюджетов.
- Split одной операции на несколько категорий.
- AI-классификация как основной механизм.
- Telegram-бот.
- Двухфакторная авторизация.
- Полная ролевая модель.
