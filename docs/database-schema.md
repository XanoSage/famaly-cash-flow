# Database Schema

Этот документ фиксирует текущую схему БД для MVP и объясняет, зачем нужна каждая таблица.

## Почему схема разделена на таблицы

Мы не храним выписку одним большим списком строк, потому что тогда сложно строить аналитику и исправлять данные. Вместо этого транзакции позже будут ссылаться на отдельные сущности: семью, счет, карту, магазин, категорию и правило категоризации. Это дает три плюса:

- можно смотреть статистику по магазинам, категориям и источникам денег;
- можно исправить категорию один раз и использовать это как правило для будущих импортов;
- можно сохранять исходные банковские данные отдельно от нашей аналитической интерпретации.

## Core Tables

### `families`

Семейный профиль. Даже если в MVP системой пользуешься ты и жена, все данные должны иметь общий корневой контекст. Это пригодится для прав доступа, семейных настроек, бюджетов и будущего Telegram bot.

### `users`

Пользователи внутри семьи. Нужны для входа, персональных действий и понимания, кто загрузил выписку или изменил категорию.

### `user_preferences`

Настройки пользователя. Сейчас главный смысл - язык интерфейса: русский для тебя и украинский для жены. Мы вынесли это отдельно, чтобы не раздувать таблицу пользователей техническими настройками.

### `accounts`

Источник денег: карточный счет, наличка, позже FOP или другие счета. Важно: счет отвечает на вопрос "где лежат деньги".

### `payment_instruments`

Физическая карта, виртуальная карта или другой инструмент доступа к счету. Это нужно из-за твоего кейса со сдвоенными картами: две карты могут относиться к одному счету, и мы не должны считать их отдельными деньгами.

### `categories`

Верхний уровень аналитики: еда, дети, здоровье, дом, спорт, накопления и так далее. `family_id` может быть пустым для системных категорий, а позже позволит добавлять семейные категории.

### `subcategories`

Более точная детализация внутри категории: например, `Еда -> Супермаркеты`, `Дети -> Школа`, `Дети -> Спорт`. Это нужно, чтобы графики были не слишком грубыми.

### `merchants`

Магазины, сервисы, организации или получатели переводов. Отдельная таблица нужна для статистики вроде "в каком супермаркете чаще покупаем и сколько тратим".

### `categorization_rules`

Правила автокатегоризации. Например: если merchant похож на `ATB`, ставить `Еда -> Супермаркеты`; если назначение содержит школу, ставить `Дети -> Школа`. Это основа для обучения системы на ручных исправлениях.

## Что будет дальше

Следующий слой схемы - импорт XLSX и транзакции:

- `import_preview_rows` для предпросмотра перед подтверждением;
- `transactions` для подтвержденных движений денег;
- позже бюджеты, уведомления и audit log.
